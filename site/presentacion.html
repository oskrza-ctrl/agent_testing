LangGraph

Un grafo formal de 9 nodos que reemplaza código imperativo.

↓

↓
③ markdown_node
↓

↓

class PipelineState(TypedDict):
    mp3_path: Any
    transcript: str
    result: Any        # AnalysisResult
    error: Optional[str]

# Cada nodo recibe estado, devuelve cambios
def node(state: PipelineState) -> dict:
    if state.get("error"):
        return {}       # early return si hay error
    result = agent.run(state["transcript"])
    return {"result": result}

🛡️ Error handling

Si cualquier nodo falla → escribe state["error"] → todos los siguientes hacen early return → el MP3 no se mueve de input/.

💡 Explícalo simple

LangGraph

Como una cadena de montaje en una fábrica

Imagina una fábrica de chocolates. El cacao pasa por varias estaciones: moler, mezclar, moldear, envolver...


Cada estación hace una sola cosa. Si la estación de moldear falla, el chocolate no llega al empaque. No se desperdicia, solo se detiene.


LangGraph es esa cadena de montaje para el audio. El MP3 entra, pasa por 9 "estaciones" (agentes) y sale organizado. Si algo falla, el audio queda intacto.

🎯 Patrón Blackboard

AnalysisAgent escribe el resultado una vez. Los demás agentes (KB, Tasks, Calendar, Drive) lo leen de forma independiente. Ningún agente conoce a los otros.

AnalysisResult

Como la ficha que llena el médico

Cuando vas al médico, el doctor te revisa y llena una ficha médica: nombre, síntomas, diagnóstico, medicamento, próxima cita...


Esa ficha la leen la enfermera, el farmacéutico y el especialista — cada uno toma lo que necesita sin preguntarle al otro.


El AnalysisResult es esa ficha. GPT la llena. Luego, Google Tasks toma las tareas, Google Calendar toma las citas, y la Knowledge Base guarda todo lo demás.

Knowledge Base

Como un archivero con cajones ordenados

Imagina un archivero con cajones. El cajón de "Ideas", el de "Tareas", el de "Reuniones"...

Técnico

RAG + ChromaDB

Búsqueda semántica sobre la Knowledge Base.

¿Qué es RAG?

Retrieval Augmented Generation. El LLM responde usando fragmentos recuperados de la KB, no de su memoria. Esto evita alucinaciones.

Flujo QUERY

1. Embed la pregunta del usuario
2. Buscar los 5 chunks más similares en ChromaDB
3. Si distancia > threshold → "No tengo esa info"
4. Pasar chunks + historial → GPT-4o-mini → respuesta

Flujo CAPTURE

GPT-4o-mini clasifica la intención. Si es CAPTURE → guarda en KB → re-indexa ChromaDB (< 2 seg). El nuevo contenido es consultable de inmediato.

def query(self, message, history):
    # 1. Embed la pregunta
    emb = self._embed([message])[0]

    # 2. Buscar en ChromaDB
    results = col.query(
        query_embeddings=[emb], n_results=5
    )

    # 3. Filtrar por relevancia
    relevant = [d for d,dist in zip(docs,dists)
                if dist < THRESHOLD]

    if not relevant:
        return "No tengo esa información."

    # 4. Generar respuesta con contexto
    return self._generate(message, relevant, history)

Chunking strategy

Acumulativos: split por --- (una entrada = un chunk)
Individuales: un archivo = un chunk

💡 Explícalo simple

RAG + ChromaDB

Una biblioteca mágica que entiende lo que quieres decir

En una biblioteca normal buscas por la palabra exacta. Si buscas "automóvil" no encuentra "coche".


La biblioteca mágica (ChromaDB) entiende el significado. Busca "coche" y encuentra "automóvil", "vehículo" y "transporte".


Cuando preguntas "¿qué reuniones tuve sobre presupuesto?", el sistema encuentra todas las notas relacionadas aunque no uses esa palabra exacta.


Y si no encuentra nada relevante, dice "no sé" en vez de inventar una respuesta.

MessageHandler

Núcleo canal-agnóstico: la misma lógica sirve a todos los canales.

# Cualquier canal hace solo esto:
handler = build_message_handler(api_key)
response = handler.process_message(texto)

# CLI (chat.py)
response = handler.process_message(input("Tu: "))
print(response)

# Telegram (telegram_bot.py)
response = handler.process_message(update.message.text)
await update.message.reply_text(response)

# Futura API web
@app.post("/chat")
def chat(msg: str):
    return handler.process_message(msg)

process_message() flow

1. classify_intent(GPT) → QUERY o CAPTURE
2. QUERY → QueryAgent.chat() → RAG
3. CAPTURE → AnalysisAgent → KB + Tasks + Calendar + re-index

process_voice(file_path) flow

1. TranscriptionAgent.run(path) → transcript
2. Siempre CAPTURE (no classify)
3. Mismo pipeline que process_message CAPTURE

agent_factory.py

build_message_handler() crea e inyecta todos los agentes. Google integrations se habilitan automáticamente si existen las credenciales.

💡 Explícalo simple

MessageHandler

Como la recepción de un hotel

La recepción de un hotel atiende a todo tipo de huésped: los que llegan a pie, los que llegan en coche, los que llegan en avión...


Pero una vez adentro, el proceso es el mismo para todos: check-in, llave, habitación.


El MessageHandler es esa recepción. Da igual si el mensaje viene de la terminal, de Telegram o de una app web — la lógica de clasificar, guardar y responder es exactamente la misma.

# Security: whitelist por user ID
ALLOWED_USER_ID = int(os.getenv("TELEGRAM_ALLOWED_USER_ID"))
def is_authorized(update):
    return update.effective_user.id == ALLOWED_USER_ID

🔄 Polling vs Webhook

Polling: el bot pregunta periódicamente si hay mensajes nuevos. No necesita URL pública. Ideal para desarrollo local.

Webhook (futuro): Telegram empuja los mensajes al servidor. Requiere URL pública (Cloud Run, paso 17).
